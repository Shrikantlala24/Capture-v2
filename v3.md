# Capture v2 - V3 Updates

This document outlines the key updates and new features introduced in Capture v2 - V3.

## UI Updations

The user interface has undergone significant enhancements to improve user experience and visual appeal. Key updates include:

*   **Modernized Design Language:** A refreshed aesthetic with cleaner layouts, updated typography, and a more intuitive navigation system.
*   **Improved Responsiveness:** The UI is now fully responsive, ensuring a seamless experience across various devices and screen sizes.
*   **Enhanced Accessibility:** Accessibility features have been improved to cater to a wider range of users.
*   **New Components and Widgets:** Introduction of new UI components for richer data representation and interactive elements.

## API Handling

The API handling mechanisms have been refined for better performance, reliability, and security.

*   **Optimized Request/Response Flow:** Improvements in how API requests are processed and responses are delivered, leading to faster data retrieval.
*   **Robust Error Handling:** Enhanced error handling and logging for API calls, providing more informative feedback and easier debugging.
*   **Security Enhancements:** Implementation of additional security measures to protect API endpoints and data.

## Langchain MCP & Gemini Model Integration

Capture v2 - V3 now leverages Langchain's Multi-Cloud Platform (MCP) for advanced language model integration, specifically with Gemini models. This integration enables sophisticated text generation, understanding, and processing capabilities.

### Gemini Models and Rate Limits

To ensure efficient and uninterrupted service, the system intelligently utilizes various Gemini models, respecting their API rate limits. The following models are integrated with their respective limitations:

*   **Gemini 2.5 Flash**
    *   Requests per Minute: 2
    *   Tokens per Minute: 36.35K
    *   Requests per Day: 5
*   **Gemini 2.5 Flash Lite**
    *   Requests per Minute: 9
    *   Tokens per Minute: 72.01K
    *   Requests per Day: 20
*   **Gemini 3 Flash**
    *   Requests per Minute: 4
    *   Tokens per Minute: 30.69K
    *   Requests per Day: 7
*   **Gemini 3.1 Flash Lite**
    *   Requests per Minute: 0
    *   Tokens per Minute: 0
    *   Requests per Day: 0 (Note: This model is currently listed with zero limits, indicating it might not be available or has very restricted access. The system will avoid using it if limits are genuinely zero.)

### Model Utilization Logic

The backend implements a sophisticated logic to circulate these models effectively, ensuring that API rate limits are not hit while optimizing for performance and response quality. This includes:

1.  **Dynamic Model Selection:** The system dynamically selects the most appropriate Gemini model based on current usage, available rate limits, and the complexity/priority of the request.
2.  **Rate Limit Monitoring:** Real-time monitoring of API usage against defined rate limits for each model.
3.  **Intelligent Fallback:** If a preferred model is nearing its rate limit, the system gracefully falls back to an available model with higher capacity.
4.  **Prioritization:** A balance is maintained between utilizing the "best" available models for higher quality responses and distributing the load across all models to prevent limit breaches.

### .env Configuration for Models

The exact names of the models to be used in the `.env` file for `MODEL_NAME` configuration will follow a specific naming convention, which will be detailed in the backend implementation documentation. An example might look like:

```
MODEL_NAME_DEFAULT=gemini-2.5-flash
MODEL_NAME_FALLBACK_1=gemini-2.5-flash-lite
# etc.
```

Further details on the backend implementation of API limiting and model utilization logic will be provided separately.
