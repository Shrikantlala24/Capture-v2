# Capture v2 - V3 Updates

This document outlines the key updates and new features introduced in Capture v2 - V3, reflecting a significant leap from the previous version.

## What's New in V3

Previously, the system simply accepted a problem statement (and optional code), providing basic logical approaches and time complexity analysis in return. 

In V3, we have significantly upgraded:

*   **Advanced Response Structure:** The output is now structured and comprehensive, providing detailed insights like:
    *   Intuition Hooks & Pattern Identification.
    *   Categorized Approach Ladders (Brute Force, Better, Optimal) with hints, pseudocode, and core logic blocks.
    *   Detailed Complexity breakdowns (Best/Average/Worst-case).
    *   Enhanced Error Diagnosis for submitted code.
    *   Curated Similar Problems to foster continued learning.
*   **Graduated Problem Discovery:** Instead of just asking for a solution, V3 allows you to discover the problem's nature and optimal approaches yourself through graduated interaction modes.
*   **Refined Prompting & Backend:** The underlying logic and prompting architecture have been completely overhauled to support this enriched analytical capability.
*   **UI Modernization:** The interface (implemented in `streamlit_monolith.py`) has been fully redesigned for clarity, responsiveness, and a more intuitive user experience.

## UI Updations

The user interface has undergone significant enhancements to improve user experience and visual appeal. Key updates include:

*   **Modernized Design Language:** A refreshed aesthetic with cleaner layouts, updated typography, and a more intuitive navigation system.
*   **Improved Responsiveness:** The UI is now fully responsive, ensuring a seamless experience across various devices and screen sizes.
*   **Enhanced Accessibility:** Accessibility features have been improved to cater to a wider range of users.
*   **New Components and Widgets:** Introduction of new UI components (mode cards, structured expanders, approach tabs) for richer data representation and interactive elements.

## API Handling

The API handling mechanisms have been refined for better performance, reliability, and security.

*   **Optimized Request/Response Flow:** Improvements in how API requests are processed and responses are delivered, leading to faster data retrieval.
*   **Robust Error Handling:** Enhanced error handling and logging for API calls, providing more informative feedback and easier debugging.
*   **Security Enhancements:** Implementation of additional security measures to protect API endpoints and data.
*   **Future-Ready Architecture:** The UI is being refactored to decouple logic from the monolith, preparing it for a clean transition to a dedicated API-driven architecture.

## Langchain MCP & Gemini Model Integration

Capture v2 - V3 now leverages Langchain's Multi-Cloud Platform (MCP) for advanced language model integration, specifically with Gemini models. This integration enables sophisticated text generation, understanding, and processing capabilities.

### Gemini Models and Rate Limits

To ensure efficient and uninterrupted service, the system intelligently utilizes various Gemini models, respecting their API rate limits. The following models are integrated with their respective limitations:

*   **Gemini 2.5 Flash** (RPM: 2, TPM: 36.35K, RPD: 5)
*   **Gemini 2.5 Flash Lite** (RPM: 9, TPM: 72.01K, RPD: 20)
*   **Gemini 3 Flash** (RPM: 4, TPM: 30.69K, RPD: 7)
*   **Gemini 3.1 Flash Lite** (Not currently utilized)

### Model Utilization Logic

The backend (`app/llm_manager.py`) implements a sophisticated logic to circulate these models effectively, ensuring that API rate limits are not hit while optimizing for performance and response quality:

1.  **Dynamic Model Selection:** Automatically selects the most appropriate model based on availability and request requirements.
2.  **Rate Limit Monitoring:** Real-time tracking of usage metrics (RPM, TPM, RPD) for each model.
3.  **Intelligent Fallback:** Automatically switches to an available high-capacity model if the preferred one nears its limit.

### .env Configuration

You must define model-specific API keys in your `.env` file to enable the new backend logic:

```env
GEMINI_2_5_FLASH_API_KEY=your_key_here
GEMINI_2_5_FLASH_LITE_API_KEY=your_key_here
GEMINI_3_FLASH_API_KEY=your_key_here
GEMINI_3_1_FLASH_LITE_API_KEY=your_key_here
```
